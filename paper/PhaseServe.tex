\begin{abstract}
Large language model (LLM) serving must satisfy both time to first token (TTFT) and time per output token (TPOT) objectives under highly heterogeneous request lengths. Phase-disaggregated systems reduce direct interference by separating prefill and decode onto different instances, but the two phases remain coupled through context queues, bridge queues, first-token delay, decode backlog, KV-block availability, and swap feasibility. We present PhaseServe, a pressure-budgeted scheduler for phase-disaggregated LLM serving. PhaseServe introduces a Pressure-Budget Controller (PBC) that converts runtime pressure into typed budgets and bottleneck-regime ownership, Budgeted Prefill Shaping (BPS) that constructs known-size prefill batches under downstream pressure constraints, and KV-Feasible Active-Set Scheduling (KAS) that selects decode requests only after hard KV and swap feasibility gates. We implement PhaseServe on a DistServe-style disaggregated runtime and evaluate it on representative chat and long-context workloads with OPT and LLaMA-family models. The results show that PhaseServe improves both first-token and generation-stage latency over the DistServe-style baseline in the measured pressure windows, while component ablations isolate the complementary roles of PBC, BPS, and KAS.
\end{abstract}

\section{Introduction}
\IEEEPARstart{L}{arge} Language Models (LLMs), such as LLaMA \cite{b1}, GPT \cite{b2}, and OPT \cite{b3}, are rapidly becoming the serving backend of interactive AI applications, including conversational assistants \cite{b11,b12}, code assistants \cite{b14}, and enterprise copilots. Unlike conventional inference services, LLM serving must sustain high throughput while meeting latency objectives under highly variable request mixes. In practice, users care about both \emph{time to first token} (TTFT), which determines how quickly the system starts responding, and \emph{time per output token} (TPOT), which determines how responsive generation remains once decoding begins. Meeting both targets efficiently is difficult because LLM inference has two phases with different resource profiles and real workloads have highly skewed input and output lengths.

Modern LLM serving systems have improved performance through continuous batching, memory-efficient KV-cache management, chunked prefill, and distributed execution \cite{orca,vllm,sarathi,tensorrt,flexgen,alpaserve}. These techniques are essential, but they still face the prefill--decode tension. Prefill processes the full prompt and is predominantly compute-intensive, whereas decode generates tokens iteratively and is typically constrained by memory traffic, KV-cache residency, and token-level queuing. When the two phases share the same serving instance, long prefills can delay latency-sensitive decode iterations, and aggressive decode batching can consume memory needed by newly admitted requests.

Phase-disaggregated serving systems address this direct interference by separating prefill and decode onto different instances \cite{b5,distserve,windserve,shuffleinfer}. This architectural split is an important step because it enables phase-specific provisioning and reduces co-location contention. Recent systems further explore dynamic dispatch, rescheduling, streaming, mixed-workload scheduling, and KV-cache transfer for serving resilience \cite{b6}. However, disaggregation does not remove cross-phase coupling; it changes its form. A prefill scheduler can still inject a burst of context-completed requests, KV footprint, and first-decode work into the bridge and decode side. A decode scheduler can still decide whether bridge-completed requests receive their first decode step promptly, whether resident requests keep KV locality, and whether swap pressure grows. We call this remaining bottleneck \emph{runtime pressure propagation}: scheduling actions in one phase alter the pressure seen by the other phase through bridge queues, first-token delay, decode backlog, KV-block scarcity, and swap feasibility.

Pressure propagation makes independent phase-local policies insufficient. Prefill has useful size information before execution: prompt length is known and strongly predicts compute and KV footprint. This favors size-aware and budget-aware batch shaping, but a prefill policy that only maximizes local utilization may flood the downstream decode side. Decode has the opposite information structure: each iteration is small, but the final output length is unknown and only revealed online. This favors attained-service or fairness-aware scheduling, but a decode policy that ignores KV residency and swap feasibility can increase stalls or delay first-token progress. The core scheduling problem in a disaggregated runtime is therefore not only to pick a good local order within each phase, but to decide which phase-local action space should be expanded or constrained under the current pressure regime.

We present \textbf{PhaseServe}, a pressure-budgeted phase scheduler for disaggregated LLM serving. PhaseServe keeps the DistServe-style split between prefill and decode, but replaces FCFS runtime scheduling with a coordinated control loop. The \emph{Pressure-Budget Controller} (PBC) observes context, bridge, first-token, decode, KV, and swap pressure, classifies the current bottleneck regime, and emits typed budgets. \emph{Budgeted Prefill Shaping} (BPS) uses these budgets to form known-size prefill batches that balance token fill, padding waste, block risk, and bounded progress. \emph{KV-Feasible Active-Set Scheduling} (KAS) uses the same budget object to select decode requests only after hard KV and swap feasibility gates, then applies attained-service, residency, age, and pressure-aware utility within the feasible set. Together, these components turn phase disaggregation from a static architectural split into an online pressure-budgeted scheduling method.

We evaluate PhaseServe on representative chat and long-context workloads with OPT and LLaMA-family models \cite{b1,b3,sharegpt,longbench}. The evaluation compares PhaseServe with DistServe-style phase-disaggregated scheduling on matched request traces and reports percentile-level TTFT and TPOT gains separately for each model, dataset, and request-rate window. This evaluation structure is important because prefill and decode pressure emerge in different operating regions; a single global speedup would obscure which bottleneck is being improved. Component ablations further show that PBC, BPS, and KAS contribute complementary benefits: PBC coordinates pressure ownership, BPS improves prefill-side first-token behavior, and KAS improves decode-side active-set selection under KV and swap constraints.

In summary, this paper makes the following contributions:
\begin{itemize}
\item We identify \emph{runtime pressure propagation} as a remaining bottleneck in phase-disaggregated LLM serving: prefill and decode are physically separated, but their scheduling actions remain coupled through context queues, bridge queues, first-token delay, decode backlog, KV-block scarcity, and swap feasibility.
\item We formulate \emph{pressure-budgeted phase scheduling}, which converts runtime pressure into typed budgets and bottleneck-regime ownership before local prefill or decode scheduling decisions are made.
\item We design and implement \textbf{PhaseServe} with three coupled components: PBC for pressure-to-budget control, BPS for budgeted known-size prefill shaping, and KAS for KV-feasible decode active-set shaping.
\item We evaluate PhaseServe against a DistServe-style baseline and component ablations, showing that the full PBC/BPS/KAS design improves TTFT, TPOT, and service attainment in the measured pressure windows.
\end{itemize}

\section{Background and Motivation}\label{background}

\subsection{LLM Serving and Phase Disaggregation}

Decoder-only LLMs generate text in two sequential phases \cite{transformer,b1,b2,b3}. The \emph{prefill} phase processes the full input prompt, computes hidden states for the prompt tokens, materializes the initial KV cache, and produces the logits from which generation begins. Prefill therefore exposes substantial matrix-computation parallelism, and its cost is largely determined by the known prompt length. The \emph{decode} phase then generates tokens one iteration at a time. Each decode step performs relatively little new computation, but it must read and extend the accumulated KV cache, making decode sensitive to memory bandwidth, KV residency, and token-level queuing.

This phase structure creates two latency objectives. TTFT measures how quickly a request reaches its first generated token, and is affected by prefill execution, prefill queueing, handoff delay, and the first decode step. TPOT measures the pace of subsequent token generation, and is affected by decode iteration latency, active-set selection, and KV-cache feasibility. Modern serving systems improve these objectives using continuous batching, paged KV-cache allocation, chunked prefill, and optimized kernels \cite{orca,vllm,tensorrt,sarathi,flashattention,flashattention2}. These techniques are essential, but they do not by themselves determine how runtime pressure should be assigned across the two phases.

Phase-disaggregated serving separates prefill and decode onto different instances \cite{b5,distserve,windserve,shuffleinfer}. This architecture is important because it removes direct co-location interference and enables phase-specific resource provisioning. DistServe, for example, disaggregates the two phases and co-optimizes placement and parallelism for TTFT and TPOT constraints \cite{distserve}. WindServe and ShuffleInfer further explore dynamic scheduling and mixed-workload dispatch in disaggregated serving \cite{windserve,shuffleinfer}, while related work also studies KV-cache streaming and transfer for serving resilience \cite{b6}. PhaseServe starts from this architectural progress, but focuses on the runtime scheduling problem that remains after the phases have been separated.

\subsection{Runtime Pressure Propagation}

Phase disaggregation removes one source of interference, but it does not make the phases independent. A prefill scheduler still decides how many prompt tokens and KV blocks are injected into the downstream path. After prefill completes, requests must cross a bridge or handoff queue, receive their first decode step, reserve append blocks, and remain feasible under decode-side KV and swap constraints. A decode scheduler, in turn, decides which bridge-completed requests are admitted, whether resident requests keep locality, and whether near-completion requests release KV blocks soon enough to unblock new first-token progress.

We call this coupling \emph{runtime pressure propagation}. Pressure is not a single queue length. It may appear as context queueing, bridge waiting, first-token debt, decode backlog, GPU-block scarcity, swap infeasibility, or long-wait progress debt. These pressure components interact. A locally efficient prefill batch can finish many contexts at once, but the resulting burst can increase bridge pressure and first-decode contention. A decode policy that aggressively protects low attained-service requests can improve some token latencies, but may delay first-decode admission or choose requests that are expensive to make KV-feasible. Conversely, a policy that only drains resident requests may preserve locality while allowing bridge pressure and TTFT tails to grow.

This propagation makes phase-local scheduling insufficient. A prefill-only policy can exploit known prompt lengths, but it cannot know whether the downstream decode side should accept more KV footprint. A decode-only policy can use attained service or residency, but it cannot decide whether the system should instead reduce prefill injection. A binary backpressure rule can stop admission under high load, but it does not say \emph{which} action space should be tightened: prefill token budget, block margin, bridge admission, decode scan, swap budget, or decode utility. A scheduler for phase-disaggregated serving therefore needs a typed control interface between observed pressure and phase-local action spaces.

The baseline exhibits three patterns. First, TTFT tail latency grows rapidly once the request rate enters a pressure region. This growth is not only prefill execution time; it is also queueing and handoff delay before the first decode step. Second, TPOT tail latency rises in a different shape, indicating that decode-side pressure is governed by token iteration, active-set contention, and memory feasibility rather than by the same bottleneck as TTFT. Third, the queue breakdown shows that pressure can move across the runtime path: context waiting grows as prefill becomes saturated, bridge waiting appears when completed contexts cannot be admitted smoothly to decode, and decode waiting remains part of the end-to-end path. The dominant queue component depends on workload shape: LongBench 4K concentrates typical waiting in the context queue, whereas ShareGPT exposes a stronger bridge interaction. The diagnostic counters further show that hard feasibility pressure and budget movement are not reducible to one latency curve: the long-context trace triggers stronger hard-pressure counters and a different prefill/decode budget response than the chat traces.

These observations do not imply that one fixed policy should always prioritize prefill or always prioritize decode. They imply that the scheduler must identify the current pressure regime and expose a compatible feasible set to each phase. When first-token or bridge pressure dominates, the system should protect bounded prefill progress and first-decode admission without violating hard memory constraints. When decode or swap pressure dominates, the decode active set should be shaped by KV and swap feasibility before soft priority. When several components are high, the scheduler needs an explicit conflict order so that local utility cannot override physical feasibility or bounded progress.

\subsection{Design Requirements}

The motivation above leads to four requirements for a phase-disaggregated scheduler. First, it must observe typed pressure rather than a single overload signal. Bridge pressure, first-token debt, decode backlog, KV scarcity, and swap feasibility have different causes and require different responses. Second, it must translate pressure into typed budgets that constrain phase-local action spaces. The response to high pressure may be to shrink prefill token injection, reserve more block margin, reduce swap exposure, alter decode scan depth, or change decode utility intensity. Third, it must separate hard feasibility from soft priority. A request that is attractive under a fairness or attained-service score is still a bad immediate choice if it cannot reserve KV blocks or would exceed the swap budget. Fourth, it must preserve bounded progress so that old or repeatedly deferred requests do not starve, while still respecting physical feasibility.

PhaseServe is designed around these requirements. PBC provides the pressure-to-budget control contract, BPS executes known-size prefill shaping under prefill token and block budgets, and KAS executes KV-feasible decode active-set shaping under scan, swap, bridge, and utility budgets. The next section describes this design.


















\section{PhaseServe Design}\label{design}

\subsection{Overview}

PhaseServe is a pressure-budgeted phase scheduler for phase-disaggregated LLM serving. It assumes the same high-level runtime split as DistServe: requests first execute prefill on a prefill instance, then enter a bridge or handoff path before iterative generation on a decode instance. PhaseServe does not replace cluster-level placement, model parallelism, or KV-cache architecture. Instead, it addresses the runtime control problem left after disaggregation: prefill decisions can still inject bridge, decode, KV, and swap pressure downstream, and decode-side decisions can still delay first-token progress upstream.

The central design principle is to convert runtime pressure into typed budgets before making local scheduling decisions. PhaseServe contains three components. The \emph{Pressure-Budget Controller} (PBC) observes context, bridge, first-token, decode, KV, and swap pressure, classifies the current bottleneck regime, and emits a structured budget object. \emph{Budgeted Prefill Shaping} (BPS) consumes this budget to shape known-size prefill batches. \emph{KV-Feasible Active-Set Scheduling} (KAS) consumes the same budget to select decode requests under hard memory and swap feasibility constraints.

Figure~\ref{fig:phase-overview} summarizes the control loop. A DistServe-style runtime exposes pressure propagation through the bridge and decode side. PhaseServe uses this feedback to update budgets: PBC translates pressure into a budget object and a bottleneck regime, BPS and KAS execute actions inside the resulting feasible sets, and the next scheduling epoch observes whether pressure has been released or shifted.

\subsection{Pressure-Budget Formulation}

At scheduling epoch $t$, PhaseServe observes a pressure vector
\begin{equation}
p(t)=\langle p_{\mathrm{ctx}},p_{\mathrm{bridge}},p_{\mathrm{first}},p_{\mathrm{decode}},p_{\mathrm{kv}},p_{\mathrm{swap}},p_{\mathrm{age}}\rangle .
\end{equation}
The components summarize, respectively, context queue pressure, bridge or unaccepted requests, requests waiting for their first decode step, decode backlog, GPU KV-block scarcity, swap pressure, and bounded-progress pressure. These signals are deliberately low cost: they are derived from queue length, waiting age, generated-token count, resident state, available GPU blocks, and swap feasibility.

PBC maps $p(t)$ to a structured budget vector
\begin{equation}
b(t)=\langle B_{\mathrm{pre}},M_{\mathrm{pre}},L_{\mathrm{scan}},B_{\mathrm{swap}},R_{\mathrm{bridge}},I_{\mathrm{dec}}\rangle ,
\end{equation}
where $B_{\mathrm{pre}}$ bounds prefill tokens admitted in the next context batch, $M_{\mathrm{pre}}$ reserves a prefill-side block margin for downstream decode, $L_{\mathrm{scan}}$ bounds decode candidate scanning, $B_{\mathrm{swap}}$ bounds swap-ins per iteration, $R_{\mathrm{bridge}}$ captures bridge-aware reserve and relaxed-admission state, and $I_{\mathrm{dec}}\in[0,1]$ controls the strength of decode-side attained-service and KV-aware utility. In the implementation, $R_{\mathrm{bridge}}$ is encoded through bridge pressure, first-token debt, and decode-side bridge reserve, fastlane, and completion-drain gates rather than as one standalone counter.

This formulation differs from ordinary backpressure. A binary backpressure mechanism mainly decides whether to admit more work. PBC decides which action space to shrink: prefill token injection, KV footprint, bridge admission, swap-in exposure, or decode local reordering. BPS and KAS then solve phase-local scheduling problems over feasible sets determined by the budget:
\begin{equation}
a_t=\pi_{\mathrm{phase}}(A_{\mathrm{phase}}(b(t)), U_{\mathrm{phase}}),
\end{equation}
where $\pi_{\mathrm{phase}}$ is a bounded online policy that filters infeasible actions and greedily ranks the remaining candidates. PhaseServe uses pressure-drift quantities such as pressure potential and pressure injection as diagnostics; it does not claim a formal global optimality guarantee.

\subsection{PBC: Regime-Aware Pressure-Budget Controller}

PBC is the control contract connecting the two phase-specialized schedulers. Its motivation is that local improvements can otherwise transfer pressure across phases. A prefill scheduler that only maximizes token utilization may finish context computation quickly but push too many requests into the bridge and decode queues. A decode scheduler that only follows attained-service order may select requests that are logically urgent but currently expensive to activate because of GPU-block or swap constraints. PBC coordinates these local policies so that they operate on compatible pressure objectives.

PBC runs in four steps. First, it normalizes each pressure component against a target operating range and aggregates related components into context, bridge/first-token, decode, soft memory/swap, and hard-feasibility summaries. Second, it classifies the current bottleneck regime and assigns an owner. The first-token-limited family is split into two submodes: a context-limited submode, where prefill queueing dominates and BPS is the TTFT owner, and a bridge/first-decode-limited submode, where completed contexts are waiting for decode admission and KAS is the TTFT owner. A decode-heavy regime is dominated by ready-decode backlog or soft swap cost; a KV/swap-limited regime is dominated by infeasible block allocation or swap-budget exhaustion; and a mixed regime has several pressure components close to their thresholds. Third, PBC maps the regime and normalized pressure into typed budgets. Context pressure protects bounded prefill progress without ignoring downstream budgets. Bridge or first-decode pressure increases bridge reserve and first-decode admission, and may cap additional prefill injection if the bridge remains high. Hard KV/swap pressure tightens $B_{\mathrm{pre}}$, increases $M_{\mathrm{pre}}$, lowers $L_{\mathrm{scan}}$ and $B_{\mathrm{swap}}$, and makes hard gates precede soft scheduling utility. Decode backlog raises $I_{\mathrm{dec}}$ when memory and swap pressure are safe. Finally, PBC applies hysteresis and smoothing so that small pressure oscillations do not cause frequent mode switches.

Algorithm~\ref{alg:pbc} summarizes the epoch-level controller. The algorithm is deliberately expressed as a budget update rather than as a request scheduler: PBC never directly chooses a prefill batch or a decode active set. It only decides the pressure regime, applies the fixed conflict order, and exposes the resulting feasible region to the phase-local executors.

\begin{algorithm}[t]
\caption{Pressure-Budget Controller (PBC)}
\label{alg:pbc}
\begin{algorithmic}[1]
\REQUIRE pressure vector $p(t)$, previous budget $b(t-1)$, previous regime $g(t-1)$
\ENSURE budget $b(t)$, regime $g(t)$, and owner $o(t)$
\STATE $s \leftarrow \mathrm{Normalize}(p(t))$
\STATE $h \leftarrow \max(s_{\mathrm{kv}}, s_{\mathrm{swap}})$ \COMMENT{hard feasibility pressure}
\IF{$h$ exceeds the hard-pressure threshold}
    \STATE $g \leftarrow \textsc{KvSwapLimited}$
    \STATE $o \leftarrow \textsc{SafetyGate}$
\ELSIF{$s_{\mathrm{ctx}}$ dominates bridge, first-token, and decode pressure}
    \STATE $g \leftarrow \textsc{FirstTokenLimited}$
    \STATE $o \leftarrow \textsc{BpsContextOwner}$
\ELSIF{$\max(s_{\mathrm{bridge}},s_{\mathrm{first}})$ dominates $s_{\mathrm{decode}}$}
    \STATE $g \leftarrow \textsc{FirstTokenLimited}$
    \STATE $o \leftarrow \textsc{KasFirstDecodeOwner}$
\ELSIF{$s_{\mathrm{decode}}$ dominates bridge and first-token pressure}
    \STATE $g \leftarrow \textsc{DecodeHeavy}$
    \STATE $o \leftarrow \textsc{KasDecodeOwner}$
\ELSE
    \STATE $g \leftarrow \textsc{Mixed}$
    \STATE $o \leftarrow \textsc{ConflictArbiter}$
\ENDIF
\STATE $\hat{b} \leftarrow \mathrm{BaseBudget}()$
\IF{$g=\textsc{KvSwapLimited}$}
    \STATE tighten $B_{\mathrm{pre}}$, increase $M_{\mathrm{pre}}$, reduce $L_{\mathrm{scan}}$ and $B_{\mathrm{swap}}$
\ELSIF{$o=\textsc{BpsContextOwner}$}
    \STATE preserve bounded BPS progress while respecting bridge and block budgets
\ELSIF{$o=\textsc{KasFirstDecodeOwner}$}
    \STATE reserve bridge/first-decode capacity and enable completion drain; cap prefill injection if needed
\ELSIF{$g=\textsc{DecodeHeavy}$}
    \STATE increase $I_{\mathrm{dec}}$ and keep $L_{\mathrm{scan}}$ large only when memory/swap pressure is safe
\ELSE
    \STATE blend budget updates using the pressure magnitudes in $s$
\ENDIF
\STATE apply precedence: feasibility, memory/swap safety, bounded progress, regime ownership, local utility
\STATE $b(t),g(t) \leftarrow \mathrm{SmoothAndHysteresis}(\hat{b}, b(t-1), g, g(t-1))$
\RETURN $b(t), g(t), o(t)$
\end{algorithmic}
\end{algorithm}

The key output of PBC is not only a set of numeric budgets, but also \emph{regime ownership}. When context-side first-token pressure dominates and hard KV/swap pressure has not been triggered, BPS is the primary TTFT owner: it preserves bounded prefill progress while still obeying token and block budgets. When bridge or first-decode debt dominates, KAS becomes the TTFT owner: it reserves feasible first-decode capacity, applies bridge-aware admission, and can drain near-completion resident requests to release KV footprint. In this submode, BPS is deliberately constrained from continuing to inject large context batches. When decode pressure dominates, KAS increases decode utility intensity and prioritizes TPOT-oriented active-set shaping. When KV or swap pressure dominates, hard feasibility gates override soft priority in both phases. In mixed regimes, PBC resolves conflicts through a fixed precedence order: physical feasibility, memory/swap safety, bounded progress, regime ownership, bridge/first-token coordination, and phase-local utility. Thus protected-oldest prefill or starved decode requests can receive progress only when physically feasible, and soft utility cannot bypass block or swap limits. This ordering is what makes PBC a typed pressure-to-budget controller rather than a loose collection of heuristics.

\subsection{BPS: Budgeted Prefill Shaping}

BPS is the prefill-side budget executor. Prefill has a useful information property: the prompt length is known before execution and provides a strong proxy for prefill compute and KV footprint. However, using this information without pressure coupling is unsafe. A pure shortest-prompt or bucket policy can reduce local prefill waiting while injecting too many context-finished requests into a decode side that is already constrained.

At each prefill dispatch, BPS forms a bounded candidate window from the waiting queue and always tracks the oldest request as a progress candidate. It partitions the window by prompt-length buckets, constructs one or more candidate batches from cost-compatible requests, and rejects any candidate that violates the physical batch size, $B_{\mathrm{pre}}$, $M_{\mathrm{pre}}$, or available-block constraints. Within that feasible region, BPS scores candidate batches using token fill, padding waste, estimated block risk, short-prompt priority under first-token pressure, and an oldest-progress bonus. When downstream hard pressure is low, this lets BPS pack cost-compatible prompts and keep GPU utilization high. When bridge, first-token, or hard KV/swap pressure rises, PBC changes the feasible region or the score weights, causing BPS to reduce pressure-increasing context injection rather than simply continuing to fill the largest possible batch.

BPS therefore differs from ordinary size-aware batching in two ways. First, its feasible set is pressure-coupled: the same prompt mix may be accepted when downstream pressure is low and rejected or deferred when bridge or hard memory pressure is high. Second, its progress path is bounded but not unconditional: an old request can override soft utility when feasible, but it cannot bypass hard token, batch, or block constraints. This design makes BPS the component that directly controls prompt-induced KV injection and first-token queueing behavior.

\subsection{KAS: KV-Feasible Decode Active-Set Shaping}

KAS is the decode-side budget executor. Decode has the opposite information structure from prefill: each iteration has a bounded immediate cost, but the total output length is unknown and only revealed as tokens are generated. Attained-service policies are attractive because they protect lightly served requests, but in LLM serving they are incomplete without memory feasibility. A request with low attained service may still be a bad immediate choice if it requires expensive swap-in or cannot reserve the next-token KV block.

KAS separates feasibility from priority. For each decode iteration, it first filters ready requests through hard gates: batch capacity, append-block availability, resident or swap-in feasibility, and the PBC swap budget $B_{\mathrm{swap}}$. Only after this hard filtering does KAS rank feasible requests. The ranking combines attained service, waiting age, resident preference, soft swap cost, skip-bounded fairness, and the decode utility intensity $I_{\mathrm{dec}}$ supplied by PBC. This distinction matters: a high swap cost can reduce a request's utility inside the feasible set, but hard swap or append-block infeasibility removes the request from that iteration entirely.

The decode utility intensity controls how aggressively KAS departs from ready-time order. In short-output or first-token-limited regimes, KAS can become FCFS-compatible so that decode-side reordering does not erase BPS's TTFT gains. In long-output or decode-heavy regimes, KAS restores stronger attained-service and KV-aware priority to target TPOT tail latency and short-output protection. Under bridge pressure, KAS may activate a completion-drain path: it prioritizes resident, near-completion, non-first-decode requests that can release KV footprint and unblock bridge admission. This path serves the first-token owner by freeing memory and bridge capacity, rather than applying aggressive attained-service ranking unconditionally.

Algorithm~\ref{alg:budgeted-executors} shows how BPS and KAS consume the same budget object in different action spaces. BPS uses known prompt lengths to build cost-compatible prefill batches under token and block budgets. KAS first filters decode candidates by KV and swap feasibility, then ranks only the feasible set by pressure-weighted utility.

\begin{algorithm}[t]
\caption{Budgeted Phase Executors (BPS and KAS)}
\label{alg:budgeted-executors}
\begin{algorithmic}[1]
\REQUIRE budget $b(t)$, prefill queue $Q_{\mathrm{pre}}$, ready decode queue $Q_{\mathrm{dec}}$
\ENSURE prefill batch $C_{\mathrm{pre}}$ and decode active set $C_{\mathrm{dec}}$
\STATE $W_{\mathrm{pre}} \leftarrow$ bounded window from $Q_{\mathrm{pre}}$ with the oldest request marked protected
\STATE $\mathcal{G} \leftarrow$ prompt-length buckets over $W_{\mathrm{pre}}$
\STATE $\mathcal{C}_{\mathrm{pre}} \leftarrow$ candidate batches formed from compatible buckets and the protected request
\STATE discard $C\in\mathcal{C}_{\mathrm{pre}}$ if it violates batch size, $B_{\mathrm{pre}}$, $M_{\mathrm{pre}}$, or available GPU blocks
\IF{$\mathcal{C}_{\mathrm{pre}}=\emptyset$}
    \STATE $C_{\mathrm{pre}} \leftarrow$ feasible protected singleton if it exists; otherwise $\emptyset$
\ELSE
\STATE $C_{\mathrm{pre}} \leftarrow \arg\max_{C\in\mathcal{C}_{\mathrm{pre}}} U_{\mathrm{pre}}(C;b(t))$
\ENDIF
\STATE $D \leftarrow \mathrm{Scan}(Q_{\mathrm{dec}}, L_{\mathrm{scan}})$; $F \leftarrow \emptyset$
\FORALL{$r \in D$}
    \IF{$r$ has a batch slot, append-block feasibility, and resident or swap-in feasibility within $B_{\mathrm{swap}}$}
        \STATE $F \leftarrow F \cup \{r\}$
    \ENDIF
\ENDFOR
\IF{$F=\emptyset$}
    \STATE $C_{\mathrm{dec}} \leftarrow \emptyset$; record a hard-feasibility stall
\ELSE
    \STATE $F_{\mathrm{bridge}} \leftarrow$ feasible first-decode and bridge-debt requests admitted by $R_{\mathrm{bridge}}$
    \STATE rank $F\setminus F_{\mathrm{bridge}}$ by attained service, age, resident preference, soft swap cost, skip state, and $I_{\mathrm{dec}}$
    \STATE $C_{\mathrm{dec}} \leftarrow F_{\mathrm{bridge}} \cup$ highest-ranked remaining requests up to decode batch capacity
\ENDIF
\RETURN $C_{\mathrm{pre}}, C_{\mathrm{dec}}$
\end{algorithmic}
\end{algorithm}

\subsection{Putting PBC, BPS, and KAS Together}

PhaseServe runs as a lightweight online loop. Each epoch begins by collecting pressure signals from the bridge, prefill queue, decode queue, KV allocator, and swap path. PBC updates the budget object and bottleneck regime. The next prefill dispatch invokes BPS with the current prefill token and block budgets; the next decode iteration invokes KAS with the current scan, swap, bridge-reserve, and utility budgets. After the actions execute, the runtime records pressure movement, infeasible rounds, protected progress, and selected action statistics for the next epoch.

This coupling is intentionally local. PhaseServe does not require exact output-length prediction, a global optimizer, or centralized per-iteration scheduling. BPS uses known prompt size and bounded candidate scanning. KAS uses generated-token count, residency, swap feasibility, and skip counters. PBC consumes aggregate pressure summaries and emits a compact budget object. These choices preserve the structure of a DistServe-style disaggregated runtime while replacing its FCFS runtime scheduling with pressure-aware phase-specialized control.

\subsection{Design Boundaries}

PhaseServe's claims are bounded by the control layer it implements. It is not a cluster-level placement algorithm and does not search tensor or pipeline parallelism configurations. It does not introduce a new KV-cache architecture; it uses KV residency, block availability, and swap feasibility as scheduling inputs. It also does not assume exact prediction of final output length. PhaseServe targets operating regions where pressure propagation is observable: prompt skew, decode backlog, bridge pressure, KV scarcity, or mixed first-token/decode contention. Its performance claims are therefore tied to the measured workload, deployment, pressure window, and reported latency percentiles.

\section{Implementation}\label{implementation}
PhaseServe is implemented on top of the open-source DistServe \cite{distserve} framework. The implementation preserves DistServe's phase-disaggregated execution stack and extends its runtime scheduler with PBC, BPS, and KAS. PBC maintains lightweight pressure summaries and emits typed budgets. BPS uses these budgets when constructing prefill batches, and KAS uses them when selecting decode active sets under KV and swap feasibility constraints. The system uses NCCL \cite{nccl} for tensor and pipeline parallel communication, and each serving instance is realized as a distributed inference engine built from multiple Ray \cite{ray} actors. Overall, PhaseServe keeps the deployment model of DistServe while replacing FCFS runtime scheduling with pressure-budgeted phase scheduling.

\section{Evaluation}\label{evaluation}

We evaluate whether pressure-budgeted phase scheduling improves runtime behavior after prefill and decode have already been disaggregated. The evaluation answers three questions:
\begin{itemize}
    \item Does PhaseServe reduce first-token and decode-token latency relative to a DistServe-style phase-disaggregated baseline on the same traces?
    \item Do the gains hold across chat and long-context workloads, rather than only for a single model or request mix?
    \item Are PBC, BPS, and KAS each needed in the pressure window used for the main end-to-end claim?
\end{itemize}

\subsection{Experimental Setup}

\textit{Baseline}. We compare PhaseServe with a DistServe-style baseline that preserves prefill/decode disaggregation but uses the baseline runtime scheduling policy. Both systems use the same request set, arrival trace, model, deployment topology, and metric extraction script for each run. This comparison isolates the effect of the PhaseServe runtime scheduler rather than placement, model parallelism, or workload sampling differences.

\textit{Models and workloads}. Table~\ref{tab:eval-workloads} summarizes the current evaluation matrix. We use OPT-13B on ShareGPT as the main setting because it exposes both prompt-length skew and decode-side variability. We use LLaMA2-13B on ShareGPT to check cross-model behavior on a chat workload, and LLaMA2-13B on LongBench 4K \cite{longbench} to stress long-context prefill pressure. Each setting uses two seeds. The seed controls the sampled request set and the generated arrival trace.

Figure~\ref{fig:stage4p-ablation} evaluates whether PBC, BPS, and KAS are jointly needed. The ablation uses OPT-13B + ShareGPT, the same seed0/seed1 request traces as the main OPT experiment, and per-GPU rates from 0.75 to 2.50 req/s/GPU. The full PhaseServe configuration improves SLO attainment, TTFT p90, and TPOT p90 over every component ablation at every measured rate in this window.

Removing PBC weakens the coordinated pressure response. Full PhaseServe improves SLO attainment over the no-PBC variant at all rates and improves both TTFT p90 and TPOT p90 throughout the window. This supports the role of PBC as more than a binary admission rule: it decides which action space should be tightened and which executor owns the response.

Removing BPS makes prefill injection less responsive to the pressure budget. The no-BPS variant loses TTFT p90 and SLO attainment relative to full PhaseServe, especially once the per-GPU rate enters the pressure region. This matches BPS's design role: it uses known prompt sizes under PBC token and block budgets, instead of only applying local FCFS ordering.

Removing KAS weakens decode-side active-set control. Full PhaseServe improves TPOT p90 over the no-KAS variant across the ablation window, while also preserving TTFT p90 gains. The result is consistent with KAS's hard-gate-first design: decode candidates must be feasible under KV and swap constraints before soft attained-service or residency utility is applied.

The ablation also exposes a boundary. TPOT p99 is not uniformly better than every ablation at every rate, particularly under mixed first-token and decode-tail pressure. We therefore use TPOT p90/p95 and SLO/TTFT tails as the main evidence for robust component benefit, and report TPOT p99 as a tradeoff-sensitive secondary metric.

\subsection{Summary of Findings}

The evaluation supports three bounded claims. First, in selected pressure windows, PhaseServe reduces both TTFT and TPOT relative to a DistServe-style phase-disaggregated baseline on matched traces. Second, the benefit appears across OPT-13B and LLaMA2-13B and across chat and long-context workloads, although the rate window and strongest percentile differ by workload. Third, the targeted ablation shows that PBC, BPS, and KAS are complementary: PBC provides pressure ownership, BPS controls prefill injection, and KAS controls KV-feasible decode active sets. The results do not imply all-rate or all-percentile dominance; they characterize the operating regions where runtime pressure propagation is observable and where pressure-budgeted scheduling changes the feasible actions of the two phases.

\section{Related Work}\label{relatedwork}

\textbf{LLM serving systems.} Modern LLM serving systems improve throughput and latency through continuous batching, paged KV-cache management, chunked prefill, optimized kernels, and distributed execution. Orca introduced iteration-level scheduling for transformer serving \cite{orca}. vLLM uses paged attention to manage KV-cache memory efficiently \cite{vllm}. SARATHI combines chunked prefill with piggybacked decoding to improve utilization under colocated execution \cite{sarathi}. TensorRT-LLM and related runtime systems optimize kernels and execution graphs for production inference \cite{tensorrt}. These systems provide important runtime foundations, but they do not by themselves define how a phase-disaggregated runtime should translate cross-phase pressure into prefill and decode scheduling budgets.

\textbf{Prefill--decode disaggregation.} Splitwise, DistServe, and DejaVu separate prefill and decode or move KV state across serving components to reduce co-location interference and improve resource efficiency \cite{b5,distserve,b6}. DistServe further co-optimizes placement and parallelism for TTFT and TPOT objectives \cite{distserve}. WindServe and ShuffleInfer study dynamic scheduling and mixed-workload dispatch in phase-disaggregated serving \cite{windserve,shuffleinfer}. PhaseServe is complementary to this line of work: it assumes a DistServe-style disaggregated runtime and focuses on the remaining runtime pressure propagation problem. Rather than selecting placement or dispatching requests across clusters, PhaseServe maps observed context, bridge, decode, KV, and swap pressure into typed budgets consumed by the local prefill and decode schedulers.

\textbf{Scheduling for heterogeneous service demands.} Classical scheduling policies exploit known processing times, attained service, or fairness constraints to reduce queueing and tail latency \cite{pinedo1992scheduling,nuyens2008foreground}. LLM serving contains both known-size and unknown-size subproblems: prompt length is known before prefill, while final output length is revealed only during decode. PhaseServe uses this asymmetry, but does not treat the two policies as independent. BPS applies known-size shaping only within PBC's prefill token and block budgets, while KAS applies attained-service and residency preferences only after hard KV/swap feasibility gates. The key difference from ordinary size-aware batching or LAS-style scheduling is the typed pressure-to-budget interface that couples the two phase-local action spaces.

\textbf{Parallelism and kernel optimization.} Model-parallel systems and compilers select tensor, pipeline, or heterogeneous execution strategies for large models \cite{alpaserve,megatron,alpa,pope,hetegen,exegpt,helix}. Kernel-level work such as FlashAttention and FlashAttention-2 reduces attention memory traffic and improves compute efficiency \cite{flashattention,flashattention2}. POD-Attention optimizes hybrid prefill/decode attention execution \cite{pod}. PhaseServe operates above these layers. It does not replace model partitioning or attention kernels; instead, it decides which requests enter prefill and decode iterations under the current pressure budget. These optimizations can therefore be combined with PhaseServe's scheduling layer.

\section{Conclusion and Future Work}
\label{conclusion}

PhaseServe is a pressure-budgeted scheduler for phase-disaggregated LLM serving. It addresses the runtime pressure propagation that remains after prefill and decode have been separated: context queues, bridge waiting, first-token debt, decode backlog, KV-block scarcity, and swap feasibility still couple the two phases. PhaseServe converts these pressure signals into typed budgets through PBC, shapes known-size prefill work through BPS, and selects KV-feasible decode active sets through KAS.

Our evaluation shows that PhaseServe reduces TTFT and TPOT relative to a DistServe-style baseline in the measured pressure windows for OPT-13B + ShareGPT, LLaMA2-13B + ShareGPT, and LLaMA2-13B + LongBench 4K. The targeted OPT-13B + ShareGPT ablation further shows that PBC, BPS, and KAS provide complementary benefits: removing any component weakens SLO attainment, TTFT p90, or TPOT p90 in the selected window. These results support the paper's central claim that phase disaggregation needs a runtime pressure-budget interface, not only separate prefill and decode instances.

The current study is bounded in three ways. First, PhaseServe does not solve cluster-level placement or dynamic resource reallocation; it assumes a fixed phase-disaggregated deployment. Second, the evaluation reports selected pressure windows rather than all-rate dominance. Third, extreme decode-tail behavior, especially TPOT p99 under mixed regimes, remains tradeoff-sensitive. Future work should combine pressure-budgeted scheduling with adaptive placement, multi-node KV transfer, and broader workload-specific SLO-scale analysis.
